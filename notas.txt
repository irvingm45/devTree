=====================================
   Estructura del package.json
=====================================

	Las etiquetas del inicio no señalan mas que datos del proyecto generales. La etiqueta "main" le sirve
a node para saber cuál es el archivo de entrada.

	"scripts" son los scripts que se pueden ejecutar, cada uno lleva una clave como puede ser "dev" y un valor como por ejemplo "nodemon index.js". Cuando nosotros queramos ejecutar algún script con npm, simplemente hacemos
		npm run {"Clave del script"}

	"author" y "license" son el nombre del autor y la licencia.
	"dependencies" son las dependencias del proyecto.
	"devDependencies" son las dependias que se necesitan para desarrollar el proyecto, es decir, una vez que se hace deployment, no son necesarias tenerlas o descargarlas para el proeycto final

================
Puntos generales
================

	Existen dos tipos de dependencias, las dependencias para el desarrollo y para producción. La diferencia
	es que las librerias de producción no estarán disponibles cuando se hagag deployment del software,
	mientras que las herramientas de producción se instalarán en el servidor.
	
	Para instalar dependencias con ayuda de npm, podemos usar el siguiente comando:
		npm i --save-dev {"Nombre de la dependencia"}
	O también existe su equivalente:
		 npm i -D {"Nombre de la dependencia"}

	¿Cómo matar procesos en puertos?
		npx kill-port {"Puerot"}

	Si queremos usar el modo watch en algún proyecto, siempre podemos usar (de manera confiable) nodemon.
Con el modo watch nos referimos a que cada vez que haya un cambio en el código, no tengamos que parar e
iniciar denuevo el proyecto.

===========
TypeScript
===========

	Realmente TypeScript es un lenguaje de programación pero no es un lenguaje que sea soportado directamente en servidores. TS no es mas que un superconjunto de JS, es decir, el código de JS funciona si lo compilamos con TS; sin embargo,
TS se diferencia de JS por tener tipado como en C++/C o Java para las variables, retornos de funciones, parámetros, etc. 
	Algo interesante de TS, es que al no ser soportado por los servidores, se compila a JS una vez que el proyecto se pasa a producción, por lo cual, TS se convierte en una dependencia de desarrollo pero no de producción al solo estar
presente hasta el deployment. Si vemos el código en el deployment, veremos JS en lugar de TS (al menos en express).

----Express con TS-----
	Al no ser sorpotado TS en Next directamente, se hace una configuración con un archivo tsconfig.json

	// tsconfig.json //
	Lo que lleva este archivo es:
	
	La salida de la compilación "outDir".
	La ruta de donde salen los archivos a compilar "rootDir".
	La libreria que se está usando "lib".
	La librería a la que apuntamos a la que salga nuestra compilación "target".
	La forma en la que se tendrá la sintaxis "moduleResolution".
	La forma en como se compilan los modulos "module".
	Existe una forma de establecer que tan estricto sea nuestro TS con "strict".
	Para debuguear nuestra aplicación "sourceMap".
	Utilizar módulos de JS y EmmaScript Modules "esModuleInterop".
	Habilitar los archivos .dev o .ts "declaration" para ver mejor que devuelve o de qué tipo es cada cosa.
